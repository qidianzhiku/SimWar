- 本文档用于描述Sim War平台的核心用户故事，以**用户价值驱动**方式串联需求、架构、API、数据库设计和前端原型，确保所有团队成员（产品、设计、前后端、测试、AI工程、运维等）对功能需求有统一理解并高效协作【4†L1661-L1669】【4†L1683-L1691】。  
- 文档帮助产品经理确认需求范围和用户价值；帮助设计师理解用户流程和交互目标；指导前后端工程师据此开发页面、组件和状态流；指导后端工程师实现API、权限、数据模型和业务逻辑；指导AI工程师定义小模型输入输出和边界；指导测试工程师据验收标准编写测试用例；指导运维理解关键业务流程的部署和监控重点。  
- 文档同时帮助 Codex 或类似AI编程助手将每条用户故事拆解为可实现的前端/后端/数据库/API/AI/测试任务，加速自动化开发。文档也是产品验收和后续迭代规划的重要依据：每条用户故事均明确功能范围和验收标准，可直接用于验收测试。

# 7. 用户认证与权限 User Stories

### US-AUTH-001：用户登录

**用户故事**：  
作为用户，  
我希望使用账号密码登录系统，  
以便访问我有权限的课程和功能。

**优先级**：P0  
**相关角色**：学员, 教师, 企业管理员, 租户管理员, 平台管理员  
**相关模块**：认证与权限  
**相关页面 / 组件**：登录页面  
**相关 API**：Auth API - POST /login  
**相关数据对象**：User, AuthToken  
**前置条件**：用户已注册账号，账号处于激活状态；系统可用。

**主流程**：
1. 用户在登录页面输入用户名/邮箱和密码。  
2. 系统通过 Auth API 验证凭据。  
3. 如果凭据正确，系统生成 JWT 访问令牌和刷新令牌，返回给客户端。  
4. 客户端保存令牌并获取用户权限信息；用户跳转到其首页或仪表盘。

**异常流程**：
- 用户名或密码错误：显示错误提示；登录失败。  
- 账号未激活或已禁用：提示无法登录；流程终止。  
- 系统宕机或网络异常：提示错误信息；登录失败。  

**权限要求**：
- 仅已注册用户可登录；未登录用户禁止访问应用内部功能。

**输入数据**：
- 用户名/邮箱，密码。

**输出数据**：
- 登录成功：返回 `authToken`, `refreshToken` 和用户信息（包含角色、租户ID等）。  
- 登录失败：错误代码和信息。

**验收标准**：

```gherkin
Given 用户已注册且账号激活
When 用户使用正确的用户名和密码请求登录
Then 系统返回有效的授权令牌并允许访问受保护资源
```

**测试建议**：
- 单元测试：验证认证服务对正确/错误凭据的处理。  
- 集成测试：测试登录API和Token生成逻辑。  
- E2E 测试：模拟用户登录流程，检查页面跳转和令牌存储。  
- 权限测试：验证未登录用户无法访问需要授权的页面。  
- 边界测试：密码长度、特殊字符等。

**Codex 开发拆解建议**：
- 前端任务：实现登录页面及表单校验；调用Auth API并处理返回。  
- 后端任务：实现 AuthController、登录校验、JWT 生成。  
- 数据库任务：查询用户表、验证密码哈希。  
- 测试任务：编写登录接口的单元和集成测试。

**备注**：
- 支持同名多租户用户时需要校验租户上下文。

### US-AUTH-002：用户登出

**用户故事**：  
作为用户，  
我希望安全地退出登录，  
以便切换账号或保证账户安全。

**优先级**：P0  
**相关角色**：学员, 教师, 管理员  
**相关模块**：认证与权限  
**相关页面 / 组件**：任意页面（全局退出按钮）  
**相关 API**：Auth API - POST /logout  
**相关数据对象**：AuthToken  
**前置条件**：用户已登录且持有有效访问令牌。

**主流程**：
1. 用户点击退出登录按钮。  
2. 系统通过 API 撤销当前令牌（可选，或在前端清除）。  
3. 客户端删除存储的令牌信息并跳转到登录页。  

**异常流程**：
- 网络异常：提示操作失败；用户可重试。  

**权限要求**：
- 登录用户可执行；未登录用户无需登出。

**输入数据**：
- 当前访问令牌。

**输出数据**：
- 成功：HTTP 200 确认登出；客户端跳转登录页。  
- 失败：HTTP 401 或错误提示。

**验收标准**：

```gherkin
Given 用户已登录并具有有效令牌
When 用户点击退出按钮
Then 系统注销会话并返回登录页面
```

**测试建议**：
- 单元测试：检查登出逻辑（令牌失效）。  
- 集成测试：前端点击登出后接口调用和重定向流程。  
- E2E 测试：登录-登出流程测试。  

**Codex 开发拆解建议**：
- 前端任务：实现全局注销功能；清理 token；跳转登录页。  
- 后端任务：实现注销接口（可选择性维护Token黑名单）。  
- 数据库任务：无新增数据表。  
- 测试任务：编写注销接口测试。

### US-AUTH-003：Token 过期自动刷新

**用户故事**：  
作为用户，  
我希望当访问令牌过期时自动刷新令牌，  
以便保持登录状态不中断并继续访问系统。

**优先级**：P1  
**相关角色**：学员, 教师, 管理员  
**相关模块**：认证与权限  
**相关 API**：Auth API - POST /refresh-token  
**相关数据对象**：AuthToken  
**前置条件**：用户持有过期的访问令牌和有效的刷新令牌。

**主流程**：
1. 访问系统时检测到访问令牌过期。  
2. 客户端调用刷新令牌接口提供 `refreshToken`。  
3. 系统验证刷新令牌有效后返回新的 `authToken` 和 `refreshToken`。  
4. 客户端保存新令牌，继续原请求或重试。

**异常流程**：
- 刷新令牌无效或过期：要求用户重新登录。  
- 网络异常：提示失败，用户可手动刷新或重新登录。

**权限要求**：
- 仅登录状态下才能刷新令牌。  
- 单租户系统要求刷新令牌携带租户ID。

**输入数据**：
- 旧的刷新令牌。

**输出数据**：
- 成功：新访问令牌和刷新令牌。  
- 失败：401错误或提示需重新登录。

**验收标准**：

```gherkin
Given 用户具有有效的刷新令牌
When 访问令牌过期且调用刷新接口
Then 系统返回新的有效访问令牌
```

**测试建议**：
- 集成测试：模拟令牌过期后刷新逻辑。  
- E2E测试：确保前端在令牌过期后自动获取新令牌并继续访问。

**Codex 开发拆解建议**：
- 后端任务：实现刷新令牌逻辑，验证刷新令牌并生成新令牌。  
- 前端任务：在拦截器中检测令牌过期并调用刷新API。  
- 测试任务：令牌刷新流程的自动化测试。

**Codex 开发拆解建议**：

```gherkin
Given 用户无权访问某资源

```

**测试建议**：
- 权限测试：用不同角色访问不同API，检查401/403响应。  
- 安全测试：验证无授权用户无法调用敏感接口。

**Codex 开发拆解建议**：
- 后端任务：在中间件中统一检查权限；实现401/403返回；记录审计日志。

### US-DECISION-005：学员提交正式决策

# 14. 回合结算 User Stories

### US-SETTLEMENT-001：教师触发回合结算

**用户故事**：  
作为教师，  
我希望在回合锁定后触发结算过程，  
以便计算当前回合的结果。

**优先级**：P0  
**相关角色**：教师  
**相关模块**：结算  
**相关 API**：Settlement API - POST /runs/{runId}/rounds/{roundId}/settle  
**相关数据对象**：StateSnapshot, SettlementResult  
**前置条件**：回合状态为 `locked`；所有队伍已提交决策（或强制）。 

**主流程**：
1. 教师点击“结算”按钮。  
2. 系统读取已冻结的 ParameterSet 和所有队伍的最新决策版本。  
3. 系统调用计量引擎接口，执行仿真计算流程（市场->运营->财务->评分）。  
4. 生成本回合的 `StateSnapshot` 和 `SettlementResult`，并将回合状态更新为 `settling` 后 `settled`。  
5. 通知教师结算完成。

**异常流程**：
- 结算失败：返回错误，允许教师重试或查看错误日志。  
- 数据不一致：日志审计，视同异常处理。

**权限要求**：
- 只有教师可触发结算；AI 不能参与正式结果写入；Plugin 只能作为调整输入，不可写入结果。【architecture constraints】

**输入数据**：
- ParameterSet ID, 各队伍决策数据。

**输出数据**：
- 新的 StateSnapshot 和 SettlementResult 数据表记录。

**验收标准**：

```gherkin
Given 回合已锁定
When 教师点击结算
Then 系统完成仿真计算并生成结算结果和状态快照
```

**测试建议**：
- Replay测试：同一输入结果应一致（幂等）。  
- 多轮测试：连续回合结果积累是否正确。  
- AI边界测试：验证AI不可替代正式计算。

**Codex 开发拆解建议**：
- 后端任务：实现结算服务调用计量引擎、保存结果。  
- 后端任务：冻结ParameterSet和Validation。  
- 测试任务：编写结算接口自动化测试。

### US-REPLAY-001：教师发起历史回合 Replay

**相关模块**：Replay  
**相关 API**：Replay API - POST /replay  
**前置条件**：选择历史已完成的回合ID。

### US-REPLAY-002：系统使用历史参数复算

**用户故事**：  
作为系统，

**优先级**：P0  

### US-REPLAY-003：系统生成 ReplayReport

**用户故事**：  
作为系统，  
我希望生成Replay后的报告，包括差异对比，  
以帮助教师分析结果一致性。

**优先级**：P0  

### US-REPLAY-004：Replay 不覆盖正式结果

**用户故事**：  
作为系统，  
我希望确保Replay过程不影响正式结果数据，  
以保持正式记录不被误改。

**优先级**：P0  
**备注**：ReplayReport追加写入，不覆盖 SettlementResult。

### US-REPLAY-005：治理人员发起 ParameterSet Shadow Replay

**用户故事**：  
作为模型治理人员，  
我希望发起使用历史回合和新参数集的 Shadow Replay，  
以验证参数变更不会造成过大偏差。

**优先级**：P1  

### US-REPLAY-006：治理人员发起 PluginPackage Shadow Replay

**用户故事**：  
作为模型治理人员，  
我希望发起使用新插件包的 Shadow Replay，  
以验证插件变更对结果的影响。

**优先级**：P1  

### US-REPLAY-007：治理人员发起 ModelVersion Shadow Replay

**用户故事**：  
作为模型治理人员，  
我希望发起使用新仿真模型版本的 Shadow Replay，  
以验证模型更新效果。

**优先级**：P2  

### US-REPLAY-008：差异超阈值进入人工审核

**用户故事**：  
作为系统，  
我希望当Replay结果与正式结果差异超过阈值时触发人工审核，  
以防重大偏差未经验证影响教学。

**优先级**：P1  

### US-REPLAY-009：Shadow Replay 通过后允许发布

**用户故事**：  
作为模型治理人员，  
我希望在Shadow Replay验证通过后允许发布新的参数/插件/模型，  
以完成更新流程。

**优先级**：P1  

### US-REPLAY-010：Shadow Replay 失败后退回修改

**用户故事**：  
作为模型治理人员，  
我希望在Shadow Replay结果不合格时退回更新请求，  
以便修改或选择不同方案。

**优先级**：P1  

> **注意**：Replay 使用历史运行中绑定的参数集与插件；Shadow Replay 完全隔离于正式系统；所有差异报告追加到审计日志；正式结果不可被Replay覆盖【4†L1683-L1691】【architecture constraints】。

**权限要求**：
- 仅平台管理员可创建或删除租户。

**验收标准**：

```gherkin
Given 平台管理员已登录
When 提交有效的租户信息
Then 系统创建新租户并返回ID
```

# 31. 风险与注意事项

| 风险 | 影响 | 相关用户故事 | 缓解措施 |
|---|---|---|---|
| 用户故事粒度过大 | 难以开发与测试，易遗漏细节 | 多条大故事 | 拆分为小功能点；保持单一职责 |
| 验收标准不明确 | 测试困难，功能不完整 | 部分故事缺少AC | 每个用户故事都需定义Given/When/Then |
| 权限边界遗漏 | 数据泄露或功能越权 | AUTH 系列故事 | 仔细审查多租户与角色权限【10†L59-L62】 |
| 学员看到未授权真值 | 学生端隐私泄露 | DECISION/RESULT相关 | 只展示 `state_obs`/`state_est`，严格隐藏 `state_true`【architecture constraints】 |
| AI输出被误认为结果 | AI输出混淆正式结果 | AI系列故事 | 输出标识为draft/advisory，不纳入正式流程 |
| ParameterSet被覆盖 | 运行中参数被改写 | SETTLEMENT, PARAM相关 | 锁定已审批参数集，不允许覆盖或删除 |
| Replay覆盖正式结果 | Replay意外修改数据 | REPLAY相关 | Replay/ShadowReplay均写入独立记录，不影响正式数据 |
| Plugin绕过Kernel写结果 | 数据真值被插件非法修改 | PLUGIN相关 | 插件仅返回建议，核心计算由Kernel完成【architecture constraints】 |
| 多租户数据串扰 | 数据隔离失败，安全问题 | AUTH, Admin相关 | 强制租户隔离检查【10†L59-L62】 |
| 用例与API不一致 | 前后端对接问题 | 所有模块 | 开发时严格对照API Contract文档 |
| 用例与前端状态不一致 | 交互流程混乱 | 前端相关UserStory | 确保前端状态流文档与故事对齐 |
| 测试覆盖不足 | 发布风险 | 所有高风险功能 | 制定覆盖表，增加Permission、Multi-tenant、Replay测试 |
