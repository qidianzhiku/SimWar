## 执行摘要

非功能需求（NFR）是保证系统质量属性和平台可靠性的关键文档。对于本项目（SimWar SaaS仿真平台）而言，性能、可靠性、可用性、安全性、可复算性、多租户隔离等质量属性尤为重要。这些质量属性直接关系到系统的稳定性、公平性和可信度。例如，性能指标（如延迟和吞吐量）决定用户体验；安全性需求（参照OWASP十大安全风险）指导我们防范最关键的安全威胁【17†L37-L44】；可复算性与审计可确保仿真结果可重现、可追溯，维护教学公平；多租户隔离保证不同用户群的数据和运行环境完全隔离【20†L53-L61】。  

NFR将对架构、开发、测试、部署和运维产生深远影响。例如，高可用架构和性能优化需要在设计阶段考虑，幂等性和事务管理需要在开发阶段实现，性能测试和安全测试需要在测试阶段执行，CI/CD管道要验证满足NFR指标，并在运维中通过监控和告警持续保障质量。NFR文档还作为Codex开发和代码验收的质量门禁，确保自动生成的代码遵循设计约束和规范。通过明确可度量的目标和监控指标，NFR帮助团队从整体上保证系统满足业务需求和用户期望。

## 性能需求

### NFR-PERF-001：API 响应时间

**分类**：性能  
**优先级**：P0  
**适用范围**：所有对外 REST/API 接口  
**业务风险**：API 响应缓慢会导致用户体验差，影响决策流畅度  
**需求描述**：在高并发条件下，绝大多数（95%）请求的响应时间应满足性能目标  
**目标值 / 标准**：95% 请求的 P95 响应时长 ≤ 300ms（请根据实际项目修改）【4†L75-L77】  
**测量方式**：统计负载测试中各接口响应时长分布  
**验收标准**：性能测试报告显示满足 P95 ≤300ms  
**监控指标**：API 响应时间分位数 (p50/p95/p99)、错误率  
**测试方式**：使用压力测试工具模拟高并发请求，验证响应时间  
**相关模块**：后端服务、API 网关  
**相关文档**：docs/contracts/api-contract.md / docs/architecture/system-architecture.md  
**失败处理**：超时请求返回超时错误；可配置重试机制  

### NFR-PERF-002：页面加载性能

**分类**：性能  
**优先级**：P1  
**适用范围**：教师端、学员端、管理后台前端页面  
**业务风险**：页面加载缓慢将影响用户体验和工作效率  
**需求描述**：各页面首次加载和切换操作应迅速，尤其是数据量较大时也要优化  
**目标值 / 标准**：教师端/学员端首屏加载 ≤2s，列表页/图表页 ≤500ms（请根据实际项目修改）  
**测量方式**：前端性能监控 (Lighthouse/LCP 等)  
**验收标准**：在真实网络环境测试符合上述阈值  
**监控指标**：前端 LCP、FCP、页面卡顿率  
**测试方式**：集成测试中测量真实场景加载时间  
**相关模块**：前端页面、前端路由  
**相关文档**：docs/frontend/figma-prototype-spec.md / docs/frontend/frontend-state-flow.md  
**失败处理**：适时显示加载动画或进度提示  

### NFR-PERF-003：决策提交性能

**分类**：性能  
**优先级**：P1  
**适用范围**：学员端决策提交功能  
**业务风险**：决策提交阻塞或超时会影响教学节奏  
**需求描述**：学员决策提交操作应快速返回结果且幂等，自动保存节流处理  
**目标值 / 标准**：决策提交接口 P95 延迟 ≤500ms（请根据实际项目修改）  
**测量方式**：压力测试模拟多学员同时提交  
**验收标准**：并发提交场景中满足延迟目标  
**监控指标**：提交接口响应时长、错误率  
**测试方式**：模拟高并发场景提交决策  
**相关模块**：Decision API / 前端决策模块  
**相关文档**：docs/contracts/api-contract.md / docs/frontend/frontend-state-flow.md  
**失败处理**：提示用户重试，未确认提交保留草稿  

### NFR-PERF-004：回合结算性能

**分类**：性能  
**优先级**：P0  
**适用范围**：系统后台结算引擎  
**业务风险**：结算过慢会延迟教学进度、影响实时性  
**需求描述**：在指定规模的队伍和参与者下，回合结算任务应在可接受时间内完成  
**目标值 / 标准**：单回合结算耗时 ≤30秒（请根据队伍数量、计算复杂度适当设定）  
**测量方式**：定期进行模拟回合结算并记录耗时  
**验收标准**：示例负载场景结算时间符合要求  
**监控指标**：结算任务耗时、排队长度  
**测试方式**：压力测试多队伍并发结算  
**相关模块**：Simulation Engine / 决策映射 / 结算引擎  
**相关文档**：docs/architecture/system-architecture.md / docs/architecture/parameter-set-management.md

**业务风险**：审计日志被篡改会掩盖安全事件

**目标值 / 标准**：审计日志设置只追加权限；至少保留半年以上历史；备份安全存储  
**测量方式**：定期验证审计记录完整性  
**验收标准**：无法删除/篡改旧日志  
**监控指标**：审计日志写入错误  
**测试方式**：尝试非法修改日志  
**相关模块**：AuditLog 服务、日志存储  
**相关文档**：docs/product/user-stories.md / docs/devops/monitoring-alerting.md  
**失败处理**：系统报警并锁定日志存储

## 多租户隔离需求

### NFR-TENANT-001：所有核心表包含 tenant_id

**分类**：多租户隔离  
**优先级**：P0  
**适用范围**：所有租户数据相关表  
**业务风险**：缺少租户标识可能导致数据混淆  
**需求描述**：每张与业务相关的数据库表必须包含 `tenant_id` 字段，用以区分不同租户的数据【20†L53-L61】。所有查询和数据操作以 `tenant_id` 过滤。  
**目标值 / 标准**：100% 数据表符合要求  
**测量方式**：数据库架构审查  
**验收标准**：无不含 `tenant_id` 的业务表  
**监控指标**：无  
**测试方式**：架构审计  
**相关模块**：数据库设计  
**相关文档**：docs/architecture/database-design.md  
**失败处理**：架构修正并迁移数据  

### NFR-TENANT-002：API 查询必须按 tenant_id 限制

**分类**：多租户隔离  
**优先级**：P0  
**适用范围**：所有对外/内部数据接口  
**业务风险**：查询不加限制会导致租户数据泄露  
**需求描述**：所有涉及多租户数据的查询都必须自动加入当前租户ID过滤条件。后端必须忽略无租户上下文的参数。  
**目标值 / 标准**：查询接口无法在不提供tenant_id的情况下返回数据  
**测量方式**：代码审计  
**验收标准**：越权查询被拒绝  
**监控指标**：越权尝试次数  
**测试方式**：模拟请求其他租户数据  
**相关模块**：API服务、ORM  
**相关文档**：docs/contracts/api-contract.md  
**失败处理**：拒绝请求并记录审计  

### NFR-TENANT-003：跨租户访问默认禁止

**分类**：多租户隔离  
**优先级**：P0  
**适用范围**：所有租户操作  
**业务风险**：租户间数据互相访问导致隐私和安全问题  
**需求描述**：默认情况下，任何请求不得访问或修改非本租户的数据。仅允许拥有跨租户权限的管理员经明确授权后访问。  
**目标值 / 标准**：无默认跨租户权限  
**测量方式**：权限审查  
**验收标准**：普通用户无法操作其他租户资源  
**监控指标**：越权请求告警  
**测试方式**：不同租户间尝试操作  
**相关模块**：权限管理、API服务  
**相关文档**：docs/product/user-stories.md  
**失败处理**：立即阻止并警告

## 数据一致性与完整性需求

### NFR-DATA-001：核心数据一致性

**分类**：数据一致性  
**优先级**：P0  
**适用范围**：所有核心数据交易（如决策、结算结果等）

## Replay 与可复算需求

### NFR-REPLAY-001：正式结果必须可 Replay

**分类**：Replay / 可复算  
**优先级**：P0  
**适用范围**：所有正式运行结果  
**业务风险**：结果不可复算会导致数据不可信、平台失去可信度  
**需求描述**：平台生成的每个正式运行结果都必须可在未来任意时刻重现。需记录所有影响结果的输入（参数、插件版本、随机种子等），并保存原始状态快照【12†L111-L116】。  
**目标值 / 标准**：任何历史 Run 绑定信息完整可恢复  
**测量方式**：随时能成功 replay 一个历史 Run  
**验收标准**：Replay与历史记录一致  
**监控指标**：Replay成功率  
**测试方式**：从任一历史Run发起Replay并比对结果  
**相关模块**：ReplayService、AuditLog  
**相关文档**：docs/quality/replay-shadow-replay-test-plan.md  
**失败处理**：审计并修复不足数据  

### NFR-REPLAY-002：Replay 使用历史绑定对象

**分类**：Replay / 可复算  
**优先级**：P0  
**适用范围**：Replay 机制  
**业务风险**：使用最新参数/模型替代历史数据会导致结果偏差  
**需求描述**：Replay 过程必须严格使用历史绑定的ScenarioPackage、PluginPackage、ParameterSet、引擎版本和随机种子，不允许自动切换到最新版本。  
**目标值 / 标准**：Replay调用接口仅接受历史版本ID  
**测量方式**：代码审查  
**验收标准**：Replay结果仅依赖传入的历史ID  
**监控指标**：无  
**测试方式**：检查重现记录使用的版本  
**相关模块**：ReplayService  
**相关文档**：docs/quality/replay-shadow-replay-test-plan.md  
**失败处理**：锁定版本再重试  

### NFR-REPLAY-003：Replay 不得覆盖正式结果

**分类**：Replay / 可复算  
**优先级**：P0  
**适用范围**：Replay 过程

**目标值 / 标准**：正式数据表与Replay无交互  
**测量方式**：代码审计  
**验收标准**：Replay执行后正式数据不变  
**监控指标**：无  
**测试方式**：运行Replay后比对正式表  
**相关模块**：ReplayService  
**相关文档**：docs/quality/replay-shadow-replay-test-plan.md  
**失败处理**：立即停止并恢复数据
